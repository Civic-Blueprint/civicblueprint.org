

# UX First Pass — Civic Blueprint Homepage

## Purpose

This document captures the first-pass UX direction for the Civic Blueprint homepage, including:

- homepage information architecture
- copy hierarchy
- conversion flow for Memo 01
- visual design implications from the Civic Blueprint logo / avatar
- accessibility and participation considerations

This is not final design. It is a working product / UX foundation for the first public version of the site.

---

## Core UX Objective

The homepage should do one thing extremely well:

# Get someone from “curious” → “reading the memo” in under 60 seconds.

Everything else is secondary.

The site does **not** need to feel like a startup landing page, campaign site, or think tank brochure.
It should feel like:

# “Here’s a serious claim. Test it.”

---

## Core UX Principles

### 1. Clear decision funnel
The homepage should guide users through a simple progression:

#### Layer 1 (5–10 seconds)
**What is this + why should I care?**

#### Layer 2 (15–30 seconds)
**Is this worth my time?**

#### Layer 3 (30–60 seconds)
**Okay, I’ll read the memo.**

---

### 2. Accessibility without flattening
The project must remain rigorous **without becoming socially unreachable**.

This means designing for:
- high-context systems thinkers
- serious domain experts
- ordinary readers with no prior context
- multiple generations and levels of technical familiarity

The goal is **not simplification for its own sake**.
The goal is:

# multi-layered accessibility

Where the same underlying ideas can be engaged at different depths.

---

### 3. Participation must feel possible
The website should not merely explain the project.
It should make users feel:

- invited
- capable of understanding it
- and able to contribute meaningfully

without requiring formal credentials or prior systems literacy.

## Homepage Structure (First Pass)

---

# 1. Hero Section

## Goal
Immediately answer:
- what this is
- why it matters
- what to do next

## Recommended structure

### Eyebrow
Optional small text:
> Open civic systems project

### Headline
> **Building a public blueprint for systems that are more capable, more humane, and harder to capture.**

### Subhead
> **What would actually have to change first?**

### CTA hierarchy
#### Primary CTA
# Read the memo

#### Secondary links
- What is this?
- Challenge this work

### Supporting microcopy
> Start with one real example. Decide if this is worth your time.

### Epistemic posture text
> A working framework. Not settled truth. Designed to be challenged.

---

# 2. Quick Context Strip

## Goal
Create fast orientation for users who need intuitive relevance before deeper explanation.

## Format
A 3–4 card strip immediately below the hero.

### Example cards
- Systems are failing in predictable ways
- AI is accelerating faster than governance
- These problems are connected
- This project tests whether that matters

## Notes
This is part of the project’s “grandmother test” / broad accessibility layer.

No jargon.
No repo references.
No systems language required.

---

# 3. Why This Exists

## Goal
Ground the project in recognizable reality.

## UX approach
Use a short explanatory paragraph plus a bullet list of everyday examples.

### Example framing
> Many essential systems are harder to use, slower to improve, and easier to capture than they should be.

### Example bullets
- Housing is scarce where it should be abundant
- Healthcare is rationed by price and complexity
- AI is moving faster than governance can respond
- Public trust erodes when systems cannot deliver

### Transition line
> **These are not separate problems.**

---

# 4. The Shift / Core Claim Section

## Goal
Introduce the project’s central analytical move.

## Suggested headline
> **What if these failures share the same upstream causes?**

## Supporting idea
This section introduces themes like:
- institutional capacity
- democratic accountability
- execution speed
- governance / coordination failures

This is the section where a user should begin to feel:
> “Okay, this may actually be worth reading.”

---

# 5. Memo 01 Conversion Section

## Goal
This is the homepage’s main conversion engine.

It should make the memo feel like:

# a concrete test case, not just another long document

## Suggested structure

### Left side
Short framing copy:
> Start with one concrete example.

### Right side (featured card)

#### Card title
> **Two Test Cases, One Framework**

#### Bullets
- Housing permitting (slow systems)
- AI governance (fast systems)
- Same framework applied to both

#### Supporting questions
- Where does capacity actually break?
- What changes across timescales?
- Does this framework outperform standard analysis?

#### CTA
# Read the memo

## UX note
This section should feel slightly like a:

# product demo / intellectual proof-of-concept

not just a reading list item.

---

# 6. Framework Overview

## Goal
Help users understand that this project has real structure behind it without overwhelming them.

## UX recommendation
Use modular / expandable cards rather than large blocks of copy.

### Suggested cards
- Principles
- Problem Map
- Systems Framework
- Process

Each card should include:
- a title
- 1–2 sentence explanation
- optional “Read more” action

# 7. Why This Site Exists

## Goal
Explain why this website exists separately from the GitHub repo.

## Suggested framing
> GitHub is not a good front door for most people.

This site exists to make the project:
- easier to understand
- easier to critique
- easier to contribute to

without requiring repo fluency or technical context.

## Accessibility / Participation Requirements

This emerged as a major design constraint during planning and should be treated as foundational.

### Core requirement
The project should be understandable and participatory for:
- younger and older readers
- technical and non-technical users
- high-context and low-context visitors
- contributors from different regions and lived experiences

### Design implication
The website should support multiple “depth layers” of engagement:

#### Layer 1 — Intuitive understanding
- Plain language
- Story / stakes
- “Why this matters to your life” framing

#### Layer 2 — Conceptual understanding
- Structured explanations
- Clear definitions
- Digestible framework summaries

#### Layer 3 — Technical depth
- Full docs
- systems analysis
- dependency / leverage reasoning

#### Layer 4 — Contribution layer
- guided critique
- structured prompts
- ways to share lived experience and expertise

## Important principle
# The solution is layering, not flattening.

The core ideas should remain rigorous.
The interface should become human.

## Visual Design Direction (Updated After Logo Review)

## Core decision
# The website should adapt to the logo, not the other way around.

The existing Civic Blueprint logo already points toward a stronger and more differentiated design system than the first homepage mockup fully captured.

The logo communicates:
- precision
- systems design
- infrastructure
- blueprint / technical rigor
- quiet seriousness

That should become the visual foundation of the website.

---

## Recommended Visual Identity

### Design keywords
- Blueprint
- Public infrastructure
- Systems map
- Drafting table
- Institutional clarity
- Open architecture
- Measured optimism

### Avoid
- startup gradients
- glassmorphism
- AI-futurist glow aesthetics
- political campaign visuals
- nonprofit softness
- generic policy blog design

---

- primary CTAs
- links
- active navigation
- icons
- visual emphasis

Use for:
- hover states
- borders
- diagrams
- secondary emphasis

`#F7F4EF`

Use for:
- main page background
- section surfaces
- cards when appropriate

### Structural line color
# Drafting Line Gray-Blue

`#C7D2E3`

Use for:
- dividers
- grid hints
- subtle structural lines
- section framing

---

## Typography Direction

The site should feel slightly more:

# architectural, public, and enduring

than a generic editorial landing page.

### Recommended UI / body font
# Public Sans

Strong fit because it communicates:
- readability
- public-interest infrastructure
- institutional seriousness

### Recommended heading option
One of:
- Source Serif 4
- Libre Baskerville
- or a strong refined sans depending on implementation

### Design note
Typography should support:
- seriousness
- clarity
- durability

not trendiness.

---

## Visual Language to Borrow from the Logo

This is one of the strongest opportunities for the website.

The logo already contains:
- drafting cues
- construction-line logic
- measurement / blueprint language
- technical geometry

The website should borrow this **subtly**.

### Suggested applications
- faint grid overlays in hero backgrounds
- blueprint-style divider lines
- corner marks around important cards or memo sections
- structured panel framing for featured content
- understated technical annotation motifs

## Important constraint
This should feel:
- intentional
- quiet
- engineered

Not gimmicky.
Not sci-fi.
Not dashboard-like.

---

## Structural UX / Visual Tone

The homepage should feel more:

# engineered than soft

That means leaning into:
- stronger grid alignment
- cleaner section framing
- deliberate modularity
- more visible structure
- crisp information architecture

This does **not** mean brutalism or visual harshness.
It means:

# the site should feel like it was designed by people trying to build something real

---

## Summary

### Best short articulation of the intended website feel:

# Editorial civic systems design

A blend of:
- public-interest think tank
- architectural blueprint
- serious product clarity
- open-source civic infrastructure

That is a strong and differentiated visual / UX direction for Civic Blueprint.
